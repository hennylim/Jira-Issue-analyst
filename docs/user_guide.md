# JIIA 시스템 운영 가이드

본 가이드는 JIIA(Jira Intelligent Issue Analyst) 에이전트의 워크플로우를 이해하고 활용하는 방법을 안내합니다.

## JIIA 워크플로우 개념

JIIA는 백그라운드에서 주기적으로 `config.yaml`에 설정된 JIRA와 연결하여,
자신(JIIA 봇 계정)에게 **할당(Assign)**된 상태인 `Open`, `To Do` 티켓을 모니터링합니다.

### 에이전트의 4단계 행동 원리

1. **감지 및 초기 스캔 (Detect & Scan)**
   - 자기가 담당자로 지정된 티켓을 발견하면, 제목과 본문 내용을 AI(`analyzer_bot`)에 전달합니다.
   - 정보가 불충분할 경우 다음 단계를 진행합니다. 정보가 충분하면 3단계로 건너뜁니다.
   
2. **IOP 엔지니어 협업 (질의응답)**
   - AI가 생성한 질문사항(예: "재현 단계 스크린샷이 있나요?", "오류가 발생한 정확한 시간대는 언제인가요?")을 **Jira 코멘트**로 남깁니다.
   - 티켓에 `jiia-status-waiting` 이라는 레이블을 추가하고 응답을 기다립니다.
   - IOP 엔지니어(사용자)가 답변을 코멘트로 남기면 에이전트가 이를 감지하여 다음 단계로 진입합니다.

3. **데이터 종합 분석 (Comprehensive Analysis)**
   - 주고받은 모든 대화 내용(Comment)과 본문을 종합하여 최종적으로 원인을 분석합니다.
   - 문제 요약, 증상, 추정 원인, SW 엔지니어 관점의 수정 방향을 담은 텍스트를 마크다운 기반으로 정리합니다.

4. **PDF 변환 및 SW 엔지니어 Handoff**
   - 분석 결과가 담긴 마크다운은 **PDF로 변환**되어 해당 Jira 티켓의 **첨부파일**로 업로드됩니다.
   - "분석 완료"라는 요약 코멘트를 남깁니다.
   - 구성에 따라 다음 단계 처리를 위해 담당자(Assignee)를 개발자로 변경(`handoff_assignee_account_id`)하고, `jiia-status-analyzed` 레이블을 부여하여 모니터링 대상에서 제외시킵니다.

## 로그 모니터링

Docker를 통해 실행 중이라면, JIIA의 판단 프로세스는 실시간으로 로그를 통해 확인할 수 있습니다.

```bash
cat logs/jiia_agent.log
# 혹은
docker-compose logs -f
```

로그 상에서 `[DRY-RUN:]` 태그가 붙어나온다면 구성이 `--dry-run` 모드인 것이며, 실제 티켓 정보는 변경되지 않고 있습니다.

## 티켓 초기화 (재분석)

만약 JIIA가 이미 분석완료한 티켓(`jiia-status-analyzed` 레이블을 가진 티켓)을 **다시 분석**하게 하고 싶다면,
Jira 환경에서 해당 티켓의 `jiia-status-analyzed` 레이블을 삭제한 후, 담당자를 다시 JIIA 봇 계정으로 변경해주면 즉시 JIIA가 반응하여 분석을 다시 시작합니다.
